Scenario Based Questions (Q601 - Q670)

Real-world problems, debugging scenarios, and architectural decisions.

These questions test your ability to apply your knowledge to solve practical problems. They often start with "How would you..." or "What happens if...".

Q601. You have a list of 1 million integers. How do you find the top 10 largest numbers efficiently? ðŸ”¥ Most Asked

Use a Min-Heap (PriorityQueue) of size 10.

  • Iterate through the list.
  • Add element to heap.
  • If heap size > 10, remove the smallest element (head).
  • At the end, the heap contains the top 10 largest elements.
  • Time Complexity: O(N log K), where K=10. Better than sorting O(N log N).

Q602. Your application is throwing OutOfMemoryError. How do you debug it?

  1. Analyze the Heap Dump using tools like Eclipse Memory Analyzer (MAT) or VisualVM.
  2. Check for Memory Leaks (objects that are no longer needed but referenced).
  3. Check if the Heap Size (`-Xmx`) is too small for the data volume.
  4. Look for large objects or static collections growing indefinitely.

Q603. How would you design a URL Shortener (like Bit.ly)?

  • Database: Key-Value store (e.g., Redis/Cassandra) mapping ShortID -> LongURL.
  • ShortID Generation: Base62 encoding (A-Z, a-z, 0-9) of a unique auto-incrementing ID or a hash.
  • Collision Handling: Ensure uniqueness of IDs (e.g., using Zookeeper for distributed ID generation).
  • Redirection: HTTP 301 (Permanent) or 302 (Temporary) redirect.

Q604. You have a slow SQL query. How do you optimize it?

  • Check Execution Plan (EXPLAIN).
  • Ensure proper Indexing on columns used in WHERE, JOIN, and ORDER BY.
  • Avoid `SELECT *`, fetch only needed columns.
  • Avoid wildcards at the start of LIKE (`%value`).
  • Check for N+1 problems if using ORM.

Q605. How do you handle transactions across multiple microservices?

Distributed transactions are hard (CAP theorem).

  • SAGA Pattern: Sequence of local transactions. If one fails, execute compensating transactions to undo previous steps.
  • Two-Phase Commit (2PC): Strict consistency but poor performance/availability. (Avoid if possible).

Q606. How would you implement a Rate Limiter?

  • Token Bucket Algorithm: User gets tokens at a fixed rate. Request consumes a token. If no tokens, reject.
  • Leaky Bucket: Requests enter a queue and are processed at a constant rate.
  • Use Redis to store counters with expiration for distributed rate limiting.

Q607. Your production server CPU is spiking to 100%. What do you do?

  1. Use `top` or `htop` to identify the process.
  2. Use `jstack ` to take a Thread Dump.
  3. Analyze the thread dump to find threads in `RUNNABLE` state.
  4. Check if they are stuck in infinite loops or heavy computation.
  5. Check GC logs (frequent Full GCs can cause high CPU).

Q608. How do you design a system to handle millions of requests per second?

  • Load Balancing: Distribute traffic.
  • Caching: Redis/Memcached for hot data.
  • Asynchronous Processing: Use Message Queues (Kafka) for non-blocking tasks.
  • Database Scaling: Sharding, Read Replicas.
  • CDN: For static assets.

Q609. How would you prevent a user from logging in from two different devices simultaneously?

  • Store the `sessionId` or `jwtToken` in the database against the `userId`.
  • When a user logs in, invalidate the previous session/token in the DB.
  • On every request, validate the token against the DB (or a fast cache like Redis).

Q610. You need to store passwords in the database. How do you do it securely?

  • NEVER store plain text.
  • Use a strong hashing algorithm like BCrypt, SCrypt, or Argon2.
  • Always use a unique Salt for each user to prevent Rainbow Table attacks.

Q611. How do you handle duplicate form submissions (e.g., user clicks 'Pay' twice)?

  • Frontend: Disable the button after the first click.
  • Backend: Use an Idempotency Key (unique request ID). If the server receives a request with an ID it has already processed, it returns the cached response instead of processing it again.

Q612. How would you implement a "Remember Me" feature?

  • Generate a long-lived, random, unique token.
  • Store it in a secure, HTTP-only cookie on the client.
  • Store the hash of the token in the database.
  • Validate the token from the cookie when the session expires.

Q613. You have a scheduled job that runs every 5 minutes. What if the previous job hasn't finished yet?

  • Use a lock (e.g., database lock or Redis lock) to ensure only one instance runs.
  • Or, configure the scheduler to skip execution if the previous task is still running (e.g., `@Scheduled` with fixedDelay instead of fixedRate).

Q614. How do you migrate a legacy Monolith to Microservices?

  • Strangler Fig Pattern: Gradually replace specific functionalities with new microservices.
  • Route traffic for new features to the microservice.
  • Keep the monolith for remaining features.
  • Repeat until the monolith is gone.

Q615. How do you handle configuration changes without restarting the application?

  • Use Spring Cloud Config Server with `@RefreshScope`.
  • Trigger a `/actuator/refresh` endpoint to reload properties.
  • Or use a dynamic configuration system like Consul or Etcd.

Q616. How would you design a notification system (Email/SMS/Push)?

  • Decouple the sender from the notifier using a Message Queue (Kafka/RabbitMQ).
  • Producer pushes "Send Notification" event.
  • Consumer reads event and calls 3rd party API (SendGrid, Twilio).
  • Implement retry logic for failed deliveries.

Q617. How do you ensure data consistency between Database and Cache (Redis)?

  • Cache-Aside: Read from cache. If miss, read from DB and update cache.
  • Write-Through: Write to cache and DB synchronously.
  • Write-Behind: Write to cache, update DB asynchronously (risk of data loss).
  • Invalidation: Delete from cache on DB update. Next read will re-populate.

Q618. How do you handle large file uploads (e.g., 1GB video)?

  • Do not buffer the whole file in memory. Use Streaming.
  • Upload directly to Cloud Storage (S3) using a Presigned URL generated by the backend. This saves backend bandwidth.
  • Use Multipart upload for reliability.

Q619. How do you secure a REST API?

  • Use HTTPS (TLS).
  • Authentication: JWT / OAuth2.
  • Rate Limiting: Prevent abuse.
  • Input Validation: Prevent Injection attacks.
  • CORS: Restrict origins.

Q620. A user reports that the website is loading very slowly. How do you investigate?

  • Check Network Tab in browser (Frontend issue? Large assets?).
  • Check Server Logs for errors or slow queries.
  • Check APM tools (New Relic, Datadog) for latency breakdown (DB vs Code vs External API).
  • Check Database CPU/Memory usage.

Q621. How do you implement pagination in a REST API?

  • Accept `page` and `size` parameters.
  • Use database `LIMIT` and `OFFSET`.
  • Return metadata: `totalElements`, `totalPages`, `currentPage`.
  • For large datasets, use Keyset Pagination (Seek Method) instead of Offset for better performance. (`WHERE id > last_seen_id LIMIT 10`).

Q622. How do you handle versioning of REST APIs?

  • URI Versioning: `/v1/users` (Most common).
  • Header Versioning: `Accept: application/vnd.company.v1+json`.
  • Parameter Versioning: `/users?version=1`.

Q623. How do you generate a unique ID in a distributed system?

  • UUID: Simple, but large (128-bit) and not sortable.
  • Twitter Snowflake: 64-bit ID containing Timestamp + Machine ID + Sequence. Sortable by time.
  • Database Sequence: Centralized (bottleneck).

Q624. How do you implement a "Like" feature (Counter)?

  • Direct DB update is slow (`UPDATE posts SET likes = likes + 1`).
  • Use Redis `INCR` command (Atomic, Fast).
  • Persist to DB asynchronously (Write-Behind) or periodically.

Q625. How do you design a Leaderboard (Top Players)?

  • Use Redis Sorted Set (`ZADD`, `ZRANGE`).
  • It maintains order automatically based on score.
  • Retrieving top 10 is extremely fast (`ZREVRANGE 0 9`).

Q626. How do you handle "Lost Update" problem in concurrency?

  • Optimistic Locking: Use a `version` column. Update only if version matches.
  • Pessimistic Locking: `SELECT ... FOR UPDATE`. Lock the row until transaction finishes.

Q627. How do you implement a Shopping Cart?

  • Session/Cookie: Good for guest users, lost if browser closed/cleared.
  • Database: Persistent, good for logged-in users.
  • Redis: Fast, persistent enough, good compromise. Key: `cart:userId`, Value: List of items.

Q628. How do you handle failed payments?

  • Do not fulfill the order immediately.
  • Listen to Webhooks from the Payment Gateway.
  • If success, update order status to "PAID".
  • If failed, notify user and allow retry.
  • Use a reconciliation job to check status of "PENDING" orders.

Q629. How do you implement Autocomplete / Typeahead?

  • Trie Data Structure: Efficient prefix search.
  • Elasticsearch: Use Edge N-gram tokenizer.
  • Redis: `ZPOS` (Lexicographical search).

Q630. How do you design a Chat Application (WhatsApp clone)?

  • Protocol: WebSocket (for real-time bi-directional communication).
  • Message Storage: Cassandra or HBase (High write throughput).
  • Status: Redis (Online/Offline).

Q631. What if your external API call times out?

  • Set a reasonable Timeout.
  • Implement Retries with Exponential Backoff.
  • Use Circuit Breaker to fail fast if the service is down.

Q632. How do you handle logging in a distributed system?

  • Centralized Logging (ELK Stack: Elasticsearch, Logstash, Kibana).
  • Use a Correlation ID (Trace ID) to link logs across services for a single request.

Q633. How do you test a microservice in isolation?

  • Unit Tests: Mock dependencies (Mockito).
  • Integration Tests: Use `Testcontainers` (spin up real DB/Kafka in Docker).
  • Contract Testing: (Pact) Ensure API contract is respected by consumer and provider.

Q634. How do you deploy a new feature with zero downtime?

  • Rolling Update: Update one instance at a time.
  • Blue-Green Deployment: Switch traffic to new environment.
  • Ensure database changes are backward compatible.

Q635. How do you handle "Hot Keys" (one item accessed very frequently)?

  • Local Cache: Cache the hot key in the application memory (EhCache/Guava) to avoid network calls to Redis.
  • Replication: Create more read replicas.

Q636. How do you design a system to upload images and generate thumbnails?

  • User uploads image to S3 (via Presigned URL).
  • S3 triggers an event (Lambda / SQS).
  • Worker service picks up the event, downloads image, resizes it (ImageMagick), uploads thumbnail back to S3.
  • Update DB with thumbnail URL.

Q637. How do you handle "Thundering Herd" problem?

  • Occurs when many clients try to reconnect or fetch data simultaneously (e.g., after cache expiry).
  • Solution: Add Jitter (random delay) to the retry/request interval so that requests are spread out.

Q638. How do you design a "Typeahead" search for a large dataset (e.g., Google Search)?

  • Use a Trie (Prefix Tree) data structure.
  • Store top K most searched queries at each node for fast retrieval.
  • Cache results in Redis.
  • Use a search engine like Elasticsearch for fuzzy matching.

Q639. How do you handle "Double Spending" in a payment system?

  • Use a unique Transaction ID (Idempotency Key).
  • Check if the transaction ID already exists in the database before processing.
  • Use database transactions with Pessimistic Locking or Optimistic Locking (versioning) to ensure atomic updates.

Q640. How do you design a "News Feed" (e.g., Facebook/Twitter)?

  • Pull Model (Fan-out on Load): Fetch tweets from all followees and sort by time. Good for small scale.
  • Push Model (Fan-out on Write): When a user tweets, push the ID to all followers' pre-computed feed lists (Redis). Good for read-heavy.
  • Hybrid: Push for normal users, Pull for celebrities (millions of followers).

Q641. How do you handle "Data Migration" with zero downtime?

  • Dual Write: Write to both old and new DB.
  • Backfill: Migrate old data in background.
  • Verify: Check consistency.
  • Switch Read: Read from new DB.
  • Stop Write: Stop writing to old DB.

Q642. How do you design a "Rate Limiter" for an API?

  • Use Redis to store request counts per user/IP.
  • Fixed Window: Reset counter every minute. (Edge case: burst at boundary).
  • Sliding Window Log: More accurate but memory intensive.
  • Token Bucket: Best for allowing bursts but enforcing average rate.

Q643. How do you handle "Distributed Locking"?

  • Use Redis (Redlock): Set a key with expiry (TTL) and a unique owner ID.
  • Use Zookeeper/Etcd: Create ephemeral nodes. If node exists, lock is taken.
  • Always have a timeout to prevent deadlocks if the holder crashes.

Q644. How do you design a "Notification System" that supports millions of users?

  • Use a Message Queue (Kafka) to buffer notifications.
  • Workers consume messages and send via 3rd party providers (FCM, APNS, SendGrid).
  • Store user preferences (email vs push) in DB.
  • Handle retries and dead-letter queues for failures.

Q645. How do you handle "Consistent Hashing"?

  • Used in distributed caching to minimize data movement when adding/removing nodes.
  • Map nodes and keys to a ring (0-2^32).
  • Key is assigned to the next node on the ring in clockwise direction.
  • Use Virtual Nodes to balance load evenly.

Q646. How do you design a "Web Crawler"?

  • URL Frontier: Queue of URLs to visit (Kafka/Redis).
  • Fetcher: Downloads page content.
  • Parser: Extracts links and data.
  • Dedup: Check if URL visited (Bloom Filter).
  • Politeness: Respect `robots.txt` and rate limits.

Q647. How do you handle "Database Sharding"?

  • Split data across multiple DB instances based on a Shard Key (e.g., UserID).
  • Horizontal Partitioning: Rows are distributed.
  • Challenges: Joins across shards are hard. Resharding is complex.

Q648. How do you design a "Ticket Booking System" (e.g., BookMyShow)?

  • Concurrency: Multiple users trying to book same seat.
  • Use Database Transaction with `SELECT ... FOR UPDATE` to lock the seat row.
  • Or use Redis to hold the seat for 5 mins (TTL). If payment fails, release it.

Q649. How do you handle "API Versioning"?

  • URI Path: `/v1/resource` (Easy to see).
  • Query Param: `?version=1` (Easy to default).
  • Header: `Accept: application/vnd.myapi.v1+json` (RESTful).

Q650. How do you design a "TinyURL" system?

  • Shortening: Base62 encoding of a unique ID.
  • ID Generation: Distributed ID generator (Snowflake) or DB auto-increment (with ranges).
  • Storage: Key-Value store (NoSQL) for fast lookups.

Q651. How do you handle "Circuit Breaker" pattern?

  • Prevents cascading failures.
  • Closed: Requests flow normally.
  • Open: Requests fail immediately (after threshold errors).
  • Half-Open: Allow few requests to test if service recovered.
  • Library: Resilience4j, Hystrix.

Q652. How do you design a "Leaderboard"?

  • Use Redis Sorted Sets (ZSET).
  • `ZADD leaderboard score user`
  • `ZREVRANGE leaderboard 0 9` (Top 10).
  • Extremely fast O(log N).

Q653. How do you handle "Idempotency" in API?

  • Client sends a unique `Idempotency-Key` header.
  • Server checks if key exists in DB/Cache.
  • If yes, return stored response.
  • If no, process and store response.

Q654. How do you design a "Chat System" (WhatsApp)?

  • Real-time: WebSockets / MQTT.
  • Storage: Cassandra / HBase (Write heavy).
  • Ordering: Timestamp / Sequence ID.
  • Status: Redis (Online/Last Seen).

Q655. How do you handle "Distributed Transactions"?

  • Avoid if possible.
  • Saga Pattern: Sequence of local transactions with compensating actions.
  • 2PC (Two-Phase Commit): Strong consistency but blocking (XA Transactions).

Q656. How do you design a "Autocomplete" system?

  • Trie: Prefix tree for fast lookups.
  • Caching: Cache top queries.
  • Pre-computation: Store results for common prefixes.

Q657. How do you handle "Session Management" in distributed systems?

  • Sticky Sessions: Load balancer routes to same server (Bad for scaling).
  • Distributed Cache: Store sessions in Redis/Memcached (Best).
  • Stateless (JWT): Store data in token on client side (No server state).

Q658. How do you design a "Job Scheduler"?

  • Quartz Scheduler: Java library.
  • Distributed: Use DB/Zookeeper for leader election to ensure only one node runs the job.
  • Message Queue: Push jobs to queue, workers consume.

Q659. How do you handle "Logging" in Microservices?

  • Centralized: ELK Stack (Elasticsearch, Logstash, Kibana).
  • Correlation ID: Pass a unique ID across service calls to trace requests.
  • Structured Logging: JSON format for easy parsing.

Q660. How do you design a "Metrics & Monitoring" system?

  • Pull Model: Prometheus scrapes metrics endpoints.
  • Push Model: App pushes to StatsD / Datadog.
  • Visualization: Grafana.
  • Alerting: PagerDuty / Slack.

Q661. How do you handle "Security" in Microservices?

  • Gateway: Auth at entry point (API Gateway).
  • mTLS: Mutual TLS for service-to-service encryption.
  • OAuth2/OIDC: Token propagation.

Q662. How do you design a "File Storage Service" (Dropbox)?

  • Chunking: Split large files into chunks.
  • Deduplication: Store unique chunks only (hash).
  • Metadata DB: Store file hierarchy and chunk mapping.
  • Blob Storage: S3 for chunks.

Q663. How do you handle "Database Indexing"?

  • B-Tree: Default, good for range queries.
  • Hash Index: Good for exact match.
  • Composite Index: Multi-column index (order matters).
  • Trade-off: Faster reads, slower writes.

Q664. How do you design a "Recommendation System"?

  • Collaborative Filtering: "Users like you liked this".
  • Content-Based: "You liked X, so you might like Y".
  • Hybrid: Combine both.
  • Use Graph DB (Neo4j) or ML models.

Q665. How do you handle "Cache Stampede"?

  • Locking: Only one thread recomputes cache.
  • Probabilistic Early Expiration: Refresh before actual expiry.

Q666. How do you design a "Payment Gateway"?

  • ACID: Strict consistency (Relational DB).
  • Idempotency: Prevent double charge.
  • Security: PCI-DSS compliance.
  • Reconciliation: Daily checks with bank.

Q667. How do you handle "High Availability"?

  • Redundancy: Multiple instances.
  • Failover: Auto-switch to standby.
  • Load Balancing: Health checks.
  • Multi-Region: Disaster recovery.

Q668. How do you design a "Gaming Leaderboard"?

  • Redis ZSET.
  • Score = Points.
  • Rank = Index.

Q669. How do you handle "Event Sourcing"?

  • Store state changes as a sequence of events.
  • Replay events to reconstruct state.
  • Audit trail is built-in.
  • Complex to query (CQRS often needed).

Q670. How do you design a "Ride Sharing App" (Uber)?

  • Geo-hashing: Grid system (S2/Geohash) to find nearby drivers.
  • WebSocket: Real-time location updates.
  • Matching Service: Match rider with driver.
  • State Machine: Trip status (Requested, Accepted, Started, Ended).
